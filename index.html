WidgetSwitcher

system in PC_MainInstance module: GamePad_Mapping engine: Unreal Engine 5.6.1

Description

The Widget Switcher is a state-driven UI management system built on PC_MainInstance (a Player Controller Blueprint). It controls which menu widget is active at any time, handling widget creation, destruction, pause state, input mode, cursor visibility, and — critically — gamepad focus management.

The system uses the E_MenuState enum to drive a Switch on Enum node inside ApplyMenuState. Each enum value maps to a distinct menu panel: Pause Menu, Options Menu, Controls Menu, and so on. When a state transition occurs, the current widget is removed, a new one is created and added to the viewport, and the correct input mode (Game Only vs. Game and UI) is applied.

Every widget that supports gamepad navigation follows the same pattern: on Construct, it calls BPFL_Widget::SetInitialWidgetFocus (after a short Delay) to hand D-pad focus to a default button. The Player Controller continuously runs RefreshMenuInput to detect whether the player is using mouse or gamepad and adjusts focus accordingly.

Gamepad Support — Unreal Only This system fully supports gamepad (controller) navigation within the Unreal Engine runtime. It uses UE5.6.1's Enhanced Input subsystem and the IMC_GamePad mapping context. Steam Input is not currently supported — Steam's controller remapping layer can intercept gamepad signals before the Enhanced Input subsystem receives them, causing bindings to fail or double-fire. When launching via Steam, gamepad UI navigation may not work as expected.

PC_MainInstance.ApplyMenuState

▸

The core function of the Widget Switcher. Reads CurrentMenuState and branches via a Switch on E_MenuState node. Each branch handles a full menu lifecycle: removing the previous widget (RemoveFromParent), creating a new one (CreateWidget), adding it to the viewport (AddToViewport), setting panel visibility (SetVisibility), pausing/unpausing the game (SetGamePaused), toggling the cursor, and storing LastFocusedWidget for gamepad D-pad navigation.

When transitioning to None (gameplay), it calls SetInputMode_GameOnly, hides the cursor, and unpauses.

PC_MainInstance.RefreshMenuInput

▸

Determines whether the player is using a mouse or gamepad and applies the correct focus behaviour. First checks if CurrentMenuState is not None (only runs when a menu is active). Then reads cursor movement delta — if the cursor is moving, the player is using a mouse and focus is cleared. If no cursor movement is detected and UsingGamepad is true, focus is set to LastFocusedWidget via SetUserFocus, enabling D-pad navigation.

This function is called from every menu widget that supports gamepad input — a pattern noted in the code as "this structure is in ALL the widgets within the game that require gamepad input."

BPFL_Widget.SetInitialWidgetFocus

▸

A static Blueprint Function Library helper called on every widget's Construct event (after a short Delay node to ensure the widget tree is fully built). Sets keyboard and gamepad focus to a specified target widget — typically the first navigable button on the menu. This is the entry point for D-pad navigation when a menu first opens.

PC_MainInstance (Player Controller)

The custom Player Controller that owns the Widget Switcher system. On BeginPlay it retrieves the EnhancedInputLocalPlayerSubsystem, validates it, and registers the IMC_GamePad mapping context (priority 0) and all project IMCs via RegisterInputMappingContext on the EnhancedInputUserSettings.

Usage Example — Call Flow

Below is the full node chain for a player pressing the Pause button, navigating the menu with a gamepad, then resuming gameplay.

1. Opening the Pause Menu

Input Action (Pause)
→
Set CurrentMenuState = PauseMenu
→
ApplyMenuState

When the player presses the pause input, the controller sets CurrentMenuState to PauseMenu and calls ApplyMenuState. Internally, the function creates WBP_TestMenu, adds it to the viewport, pauses the game, shows the mouse cursor, and stores ResumeButton as the LastFocusedWidget for gamepad navigation.

2. Widget Construct — Gamepad Focus

WBP_TestMenu::Construct
→
Delay (0.1s)
→
BPFL_Widget::SetInitialWidgetFocus(ResumeButton)

Every menu widget follows this identical Construct → Delay → SetInitialWidgetFocus pattern to establish gamepad focus.

3. Resuming Gameplay

When the player clicks or presses the Resume button, the widget plays a button sound, sets CurrentMenuState to None, and calls ApplyMenuState on the PC Main Instance.

On Clicked (ResumeButton) → Play Sound 2D → Set CurrentMenuState to None → ApplyMenuState on PC Main Instance.

ApplyMenuState then runs the None branch — collapsing all menu panels, removing the widget, unpausing the game, switching to Game Only input mode, and hiding the cursor:

ApplyMenuState (None branch): Set Visibility → Collapsed on all sub-panels, Remove From Parent, Set Game Paused (false), Set Input Mode Game Only, Set Show Mouse Cursor (false), Set CurrentMenuState = None.

Known Limitations

Steam Input Not Supported Gamepad controls work correctly when running the project directly through the Unreal Editor or a standalone build. However, when launched via Steam, the Steam Input system can intercept controller signals before UE5.6.1's Enhanced Input subsystem processes them. This causes gamepad UI navigation and the IMC_GamePad mapping context to malfunction. Disabling Steam Input per-game in Steam's controller settings is a current workaround, but is not a reliable end-user solution. Full Steam Input integration is not yet implemented.
LimitationDetail
Steam Input conflict Steam's overlay remapping layer intercepts raw gamepad input. IMC_GamePad bindings may not fire, and UsingGamepad may remain false even with a controller connected.
Focus race on Construct The Delay before SetInitialWidgetFocus is a fixed duration. On very slow machines, the widget tree may not be ready, causing focus to silently fail.
Single-widget stack ApplyMenuState removes the previous widget before creating the next. There is no history stack — the caller must know which state to return to (e.g., Options → Pause is hardcoded).
No hot-plug detection If a gamepad is connected after a menu is already open, UsingGamepad will update on the next RefreshMenuInput call, but LastFocusedWidget may point to a stale reference if the widget was rebuilt.
